Proactive WebSocket Connection Refresh for Railway's 15-Minute Limit

 Context

 Railway.app terminates persistent WebSocket connections after 15 minutes regardless of keepalive activity. Voice interviews routinely run 20-45+ minutes, causing forced
 mid-interview disconnections. This plan adds a proactive, transparent connection refresh at a natural conversation boundary, reusing existing reconnection scaffolding but with
 dedicated refresh semantics distinct from pause/resume.

 Design

 Trigger: Flag → Conversation Boundary

 1. Watchdog sets flag (needsConnectionRefresh) when client WS age > 13.5 min
 2. speech_stopped handler checks flag → cancels provider's auto-response, sets pendingRefreshAfterTranscript
 3. transcription.completed handler checks that flag → saves transcript, sends it to client, then triggers refresh (skips Barbara analysis)
 4. Text-input path checks flag before response.create → triggers refresh instead
 5. Fallback: Watchdog forces refresh at 14.5 min if no boundary has occurred (long silence, paused)

 Refresh vs Resume: Separate Semantics

 The existing resume_interview path sends "welcome back" language via buildResumeInstructions() (instructions.ts:180-260). A planned refresh must be invisible to the respondent —
  Alvia should respond to the last utterance naturally as if nothing happened.

 Solution: add buildRefreshInstructions() that includes the same transcript context as resume but with "Continue the conversation naturally. Respond to the respondent's last
 statement." instead of "Welcome them back." The reconnect handler detects isConnectionRefresh and uses this path, skipping awaitingResume and auto-triggering response creation
 (no mic click needed).

 Partial Audio Leakage

 Despite response.cancel at speech_stopped, audio deltas can race ahead. Client-side mitigation: when connection_refresh message is received, suppress audio playback until the
 new connected message arrives.
